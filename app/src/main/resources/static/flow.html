OAuth2 token exchange - interactive demo
Walk through each step to see how a user logs in, gets an access token, calls a protected API, and exchanges that token for a downstream service token — all within a single Spring Boot app.
Architecture
A
Authorization server - :8080/oauth2/token
Issues JWTs signed with an RSA key. Handles login, authorization code flow, client credentials, and token exchange.
B
Resource server - :8080/api/**
Validates Bearer tokens using the same local JwtDecoder. No HTTP round-trip needed — keys are in-process.
C
Token exchange - RFC 8693
The resource server can call /oauth2/token with grant type token-exchange to mint a new token scoped to a downstream service.

Step 1 — Login
The user authenticates via Spring Security's form login. On success, the Authorization Server creates an authorization code that will be exchanged for a token.
Login form — POST /login
POST /login (form submit)
Content-Type: application/x-www-form-urlencoded username=user&password=password → 302 redirect to /oauth2/authorize?... → 302 redirect to redirect_uri with ?code=AUTH_CODE
Step 2 - Get access token
The client sends the authorization code to the token endpoint. The server validates it and returns a signed JWT access token.
Token request — POST /oauth2/token
POST /oauth2/token
Authorization: Basic bXktY2xpZW50OnNlY3JldA== Content-Type: application/x-www-form-urlencoded grant_type=authorization_code &code=AUTH_CODE_FROM_REDIRECT &redirect_uri=http://localhost:8080/login/oauth2/code/my-client
Step 3 - Call protected API
The client sends the JWT as a Bearer token. The resource server validates it locally using the shared JwtDecoder and maps the roles claim to Spring Security authorities.

Step 4 — Exchange the token
The resource server acts as a client and swaps the user's token for a new one scoped to a downstream service. This is the RFC 8693 token exchange grant type.
Token exchange request
POST /oauth2/token
Authorization: Basic bXktY2xpZW50OnNlY3JldA== Content-Type: application/x-www-form-urlencoded grant_type=urn:ietf:params:oauth:grant-type:token-exchange &subject_token=<original access token> &subject_token_type=urn:ietf:params:oauth:token-type:access_token &requested_token_type=urn:ietf:params:oauth:token-type:access_token &audience=internal-service
Step 5 — Use the exchanged token
The new token is scoped to the downstream service. Inspect and compare both tokens — notice the audience and any scope differences.
What changed?
✓
Audience (aud) is now scoped to internal-service
The downstream service can reject tokens not addressed to it, limiting token reuse across services.
✓
New expiry — shorter TTL possible
Token exchange lets you issue tighter short-lived tokens for machine-to-machine hops without re-authenticating the user.
✓
Original subject (sub) preserved
The downstream service still knows which user initiated the call, enabling end-to-end audit trails.
